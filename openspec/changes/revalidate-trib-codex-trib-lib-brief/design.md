## Context

Codex Tribunal already has a dependency-free Python orchestrator, installable CLI/API, reusable skill, nine disclosed persona definitions, deterministic gates, a structured OSS matrix, and a public live-IDR report. The exact supplied brief was previously completed, but the repository is now at a closed public commit and the acceptance-critical facts remain mutable. This pass must therefore distinguish historical evidence from observations made after this proposal was created.

Four evidence classes remain separate: NotebookLM supports research synthesis; GitHub supplies primary mutable repository metadata; external models supply adversarial opinions; and executable CLI/API observations prove the local package contract. A generated assertion from one class cannot substitute for evidence from another.

## Goals / Non-Goals

**Goals:**

- Reproduce every live obligation in the brief from a new OpenSpec baseline.
- Preserve sanitized raw provenance for NotebookLM queries, Grok attempts or authorized fallbacks, GitHub metadata, and real E2E behavior.
- Reconcile contradictions against primary or executable evidence before updating the public synthesis.
- Keep the 100-point comparison mechanically consistent, differentiated, and crowned exactly once.
- Publish a pushed commit whose SHA-pinned public report blob is retrievable and contains the requested result.

**Non-Goals:**

- Claim that the bundled `local-rules` backend performs semantic fact checking, NotebookLM retrieval, visual UI inspection, or provider-family independence.
- Add a production model provider, browser UI, durable quota store, or trace platform merely to make the audit look broader.
- Present failed provider calls, NotebookLM prose, star popularity, or the synthetic Karpathy-inspired persona as independent truth.
- Rewrite stable runtime code unless the real E2E exposes a scoped defect.

## Decisions

1. **Create a new pass-specific change.** Earlier completed changes remain historical inputs. Reusing their completed task state was rejected because it would blur old and current observations.
2. **Reuse the canonical notebook, candidate set, and published rubric.** This keeps results comparable while live inventory and queries establish freshness. A second notebook or newly invented scoring system would fragment the evidence.
3. **Run independent calls from a frozen conclusion-free packet.** Each role receives the same evidence boundary and output contract without sibling verdicts or a proposed winner. Sequential debate was rejected because it weakens initial independence.
4. **Attempt the mandated Grok form first for all three roles.** Only a complete model response can become a verdict; an availability or quota failure is retained as a failed attempt and activates only the brief-authorized `agy` fallback.
5. **Refresh GitHub facts in one bounded batch.** Canonical URLs, numeric stars, archive/activity and license signals are timestamped together. Stars remain unscored context.
6. **Use consumer-facing E2E surfaces.** Source CLI, all primary modes, comparison, invalid input, PEP 517 wheel installation, installed console entry point, and installed Python API are exercised without a mocked backend.
7. **Fail closed at publication.** Unit/compile/build/skill/CSV/report/OpenSpec/evidence checks, staged-diff inspection, remote-SHA equality, and immutable HTTP retrieval all gate the final handoff.

## Risks / Trade-offs

- **Provider quota prevents a required verdict** → retain the exact sanitized failure, use only the authorized fallback, and never synthesize missing prose.
- **NotebookLM produces stale or invented implementation claims** → run a contradiction/source-attribution control and prefer current primary or executable evidence.
- **GitHub metadata drifts during the run** → capture candidates in one dated snapshot and update all dependent surfaces together.
- **A local structural run looks semantically green** → keep backend identity, score ceiling, warning marker, and evidence gaps visible in the E2E ledger and report.
- **External output leaks credentials or local details** → commit only sanitized facts and inspect the final diff for tokens, cookies, and unrelated paths.
- **No report facts changed after revalidation** → still publish a dated evidence ledger and report reference so the new live pass is externally auditable.

## Migration Plan

1. Capture the starting Git/guard/tool state and strictly validate these planning artifacts.
2. Run authenticated NotebookLM research, the GitHub batch, three isolated judge paths, and reconciliation.
3. Execute source-tree and clean-install E2E observations, then update evidence, matrix, and report consistently.
4. Run all gates, complete task checkboxes, inspect and stage only intended artifacts, commit, and push.
5. Compare local and remote full SHAs and retrieve the exact SHA-pinned public report blob.

Rollback is a normal revert of the pass-specific publication commit. Temporary prompts, model sessions, build directories, and isolated virtual environments are not public artifacts.
