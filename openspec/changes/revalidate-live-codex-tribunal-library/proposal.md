## Why

The public Codex Tribunal release needs a fresh, independently reproducible closure pass because its central claims depend on mutable external evidence, live NotebookLM research, model availability, and an installed-package E2E path. Revalidation must preserve failed-provider evidence and capability boundaries instead of silently inheriting earlier green conclusions.

## What Changes

- Revalidate the canonical authenticated NotebookLM notebook, targeted primary OSS sources, and multiple cross-source queries with a durable evidence ledger.
- Run three isolated adversarial judge perspectives for knowledge/correctness, hostile critique/risk, and UI/UX feasibility, attempting the mandated Grok path first and recording any truthful fallback provenance.
- Refresh or verify the dated GitHub metadata and the feature matrix with numeric stars, a deterministic 100-point rubric, exactly one eligible winner, and explicit category/license qualifications.
- Update the final tribunal report so its IDR declaration, three verdicts, synthesis, Mermaid flow, limitations, implementation plan, and evidence links agree with the revalidated artifacts.
- Exercise the recommendation through the repository CLI, all primary modes, expected failure behavior, a clean PEP 517 install, installed console entry point, and installed Python API without mocked model or backend results.
- Require strict OpenSpec, report, CSV, skill, test, build, commit, push, and immutable public blob checks before closure.

## Capabilities

### New Capabilities

- `live-tribunal-revalidation`: Defines the fail-closed contract for a fresh live IDR, independent tribunal perspectives, evidence-bound OSS synthesis, real E2E proof, and publication verification.

### Modified Capabilities

None.

## Impact

The change primarily affects `report/codex-trib-lib-tribunal.md`, evidence ledgers under `report/evidence/`, the machine-readable comparison snapshot/matrix when live facts have changed, and the new OpenSpec artifacts. Runtime or test changes are permitted only when the revalidation exposes a reproducible defect in the declared library/skill behavior. External systems exercised are authenticated NotebookLM, GitHub metadata, the configured Grok or approved `agy` judge path, Python packaging, Git, and the public GitHub blob surface.
