## Context

Codex Tribunal is already implemented and published as a dependency-free Python CLI/API plus reusable skill. Earlier passes contain substantial evidence, but mutable external facts and inherited prose cannot close the current brief. This pass must independently observe authenticated NotebookLM research, current primary GitHub metadata, three isolated adversarial judgments, executable package behavior, and immutable public retrieval. It must also preserve the distinction between the local structural backend, external research, external judge opinion, project-authored scoring, and direct runtime evidence.

The repository is shared and the current runtime is already covered by unit, packaging, installed CLI/API, skill, and report validators. Runtime or persona changes are therefore conditional on reproducing a scoped defect. External services may fail because of capacity, quota, authentication, or provider limits; such failures must be retained as evidence rather than silently converted into success.

## Goals / Non-Goals

**Goals:**

- Produce a self-contained, current-pass OpenSpec contract before changing the report or runtime.
- Revalidate the canonical public NotebookLM notebook and run multiple source-grounded, role-specific cross-queries without treating failed sources as evidence.
- Obtain one fresh isolated verdict for each required Tribunal perspective, preferring Grok 4.5 High and using the brief-authorized `agy` fallback only when Grok produces no usable model verdict.
- Refresh the OSS-first comparison from primary GitHub surfaces and apply one explicit 100-point rubric with numeric stars and exactly one crown.
- Exercise the recommendation through source, build, clean-install CLI, installed API, expected-error, skill, matrix, report, and publication surfaces.
- Publish a SHA-pinned report and record enough provenance to distinguish observation from interpretation.

**Non-Goals:**

- Claim that the bundled `local-rules` backend performs semantic fact checking, NotebookLM retrieval, browser/visual inspection, or production observability.
- Claim statistical independence or provider-family diversity merely because prompts, personas, or sessions differ.
- Modify stable runtime behavior, persona semantics, or package structure unless this pass reproduces a requirement-breaking defect.
- Delete or repurpose unrelated NotebookLM sources/notebooks to make room for this pass.
- Treat GitHub stars, generated prose, accepted source-add commands, or a successful push command as proof of quality, grounding, processing, or public availability.

## Decisions

1. **Use a fail-closed evidence hierarchy.** Current executable behavior and primary service responses control mutable facts; NotebookLM synthesis and external judge prose are advisory and must be manually reconciled. This is preferred over trusting any generated narrative because the corpus can contain stale repository snapshots and circular prior reports.

2. **Continue the canonical NotebookLM notebook without destructive capacity management.** The pass will authenticate, inspect sharing and source states, select only processed source IDs, and query role-specific source sets. If source or account capacity is exhausted, the failure is recorded and existing processed sources are reused. Deleting unrelated user content was rejected because the brief does not authorize destructive account changes.

3. **Freeze one conclusion-free judge packet.** The three roles receive identical factual inputs, requirements, and evidence references, but role-specific instructions and separate sessions. Grok 4.5 High is attempted exactly as required; `agy` is accepted only when Grok yields no verdict, and all accepted verdicts retain engine/session/packet provenance. Feeding one judge another judge's conclusion was rejected because it would contaminate the initial independent views.

4. **Score OSS fit with a project-authored rubric, not popularity.** Primary GitHub API responses and root documentation establish identity, stars, update time, maintenance posture, and license constraints. The 100-point score evaluates bounded use-case fit; stars receive zero rubric points. A single crown is assigned only after the full matrix validates.

5. **Keep evidence classes in separate ledgers.** NotebookLM, external judge attempts, accepted verdicts, OSS metadata, synthesis, E2E execution, and publication each receive a current-pass evidence file. The report links these artifacts and states conflicts explicitly. This is preferred over one monolithic transcript because it makes provenance and failure boundaries auditable.

6. **Change runtime only on reproduced failure.** Existing code is exercised first. If its documented source, package, API, CLI, error, or skill behavior fails, the smallest scoped fix and a regression test are added; otherwise this remains an evidence/report change. This avoids speculative refactoring during a verification task.

7. **Gate completion on immutable retrieval.** After checks pass, all intended files are force-added as requested, committed, pushed, and fetched through the public raw/blob surface at the exact commit SHA. A branch-level URL alone is insufficient because it can move after publication.

## Risks / Trade-offs

- **[NotebookLM capacity or stale synthesis]** -> Record exact source states and query metadata, exclude failed sources and circular report sources from control queries, and let current source/execution overrule generated claims.
- **[Grok quota/service failure]** -> Preserve the failed attempt, invoke only the authorized `agy` fallback in isolated sessions, and disclose any lack of provider-family diversity.
- **[Judge prompt contamination]** -> Hash one conclusion-free packet, prevent cross-role verdict sharing, and keep the final synthesis outside the initial judgments.
- **[Mutable GitHub metadata]** -> Timestamp one primary refresh, update JSON/CSV/report atomically, and validate their equality before commit.
- **[Self-awarded crown mistaken for benchmark evidence]** -> Publish rubric ownership, anti-gaming rules, dated inputs, category mismatches, and the narrow declared use case next to the crown.
- **[Passing unit tests but broken user surface]** -> Build a wheel, install it into an isolated environment, run real CLI/API/error cases, validate the reusable skill, and retrieve the public report at the final SHA.
- **[Shared-worktree interference]** -> Preserve unrelated changes, inspect the final diff, and include only files belonging to this change.
- **[Publication changes the SHA after evidence is written]** -> Keep pre-push execution evidence content-addressed where possible, then add a small publication record in a final commit and verify that exact immutable record separately.

## Migration Plan

1. Complete and validate the OpenSpec artifacts.
2. Capture fresh external evidence without changing runtime code.
3. Reconcile generated findings against source and executable behavior.
4. Update current-pass ledgers, matrix, and report; repair runtime only if a documented behavior fails.
5. Run strict source/build/install/E2E/report/OpenSpec/diff gates.
6. Commit and push the verified change, retrieve the SHA-pinned public report, and record the publication proof.

Rollback is documentation-first: if an external or executable gate fails, do not publish a success claim. If a scoped runtime fix regresses a gate, revert only that pass's own fix and retain the failing evidence. Published historical ledgers remain immutable context rather than being rewritten.
