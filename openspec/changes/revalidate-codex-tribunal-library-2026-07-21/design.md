## Context

The project already ships a dependency-free Python Tribunal core, a reusable skill, nine validated personas including a disclosed Karpathy-inspired critic, deterministic gates, a comparison matrix, and a public live-IDR report. Earlier evidence proves only earlier snapshots. This pass must re-observe mutable external systems while preserving the distinction between NotebookLM research, external judge opinion, primary GitHub metadata, project-authored comparative scoring, and executable package behavior.

The baseline weekly guard reports `weekly_used=73`, so this pass must not start expensive Codex/OMX team bursts. The brief specifically requires three Grok workers or permits `agy` perspectives. Three isolated Grok single-turn calls are attempted first; if the service returns no model output, fresh isolated `agy` role sessions satisfy the explicit fallback while the failed Grok provenance remains visible. `fast=not_seen`, so the no-fast invariant is satisfied.

## Goals / Non-Goals

**Goals:**

- Produce a fresh authenticated NotebookLM ledger with canonical identity, source state, multiple role-aligned queries, and one contradiction/source-attribution control.
- Reconfirm current OSS repository facts and numeric GitHub stars from primary GitHub surfaces while keeping the 100-point rubric reproducible.
- Freeze a conclusion-free evidence packet before three isolated role judgments, prefer Grok as requested, fall back only through the brief-authorized `agy` path, and synthesize only structurally complete outputs.
- Re-run the package through source, build, clean-install console, installed API, successful comparison, and expected-error paths.
- Publish an internally consistent report/evidence set and prove the exact committed report is public at an immutable blob URL.

**Non-Goals:**

- Claim that the deterministic `local-rules` backend performs semantic fact-checking, live browsing, visual UX inspection, or NotebookLM retrieval.
- Turn mutable stars into quality points or generated model prose into controlling factual evidence.
- Impersonate Andrej Karpathy or attribute the synthetic persona's conclusions to him.

## Decisions

1. **Reuse the historical canonical NotebookLM notebook with an explicit processed-source subset.** The notebook already contains the required OSS, usability, bias, and Tribunal sources, but it also holds exactly 600 processed sources. Two concurrent pass-specific source-adds returned IDs and then failed with empty content/status `3`; creating a fresh notebook failed because the account already has 500 notebooks. Deleting or repurposing unrelated user notebooks was rejected as destructive. This pass therefore uses only explicitly selected processed source IDs, fresh conversation IDs, a pass-specific query ledger, and current primary/executable controls. The failed adds are evidence of the capacity boundary, never cited as research.
2. **Use primary repository pages plus executable checks as controlling evidence.** NotebookLM and Grok answers remain research or review opinions. GitHub API results, repository files, CLI output, wheel contents, and exact remote/blob checks control conflicts. Treating any model answer as self-validating was rejected.
3. **Run three isolated external single-turn role paths with truthful fallback provenance.** Each Grok attempt receives the same frozen, conclusion-free packet plus one role-specific rubric and cannot see sibling output. If Grok returns no model output, retain the failed attempt and use the brief-authorized `agy` fallback in a fresh isolated session. A verdict is accepted only if it reports role, engine/model provenance, inspected evidence, score, recommendation, concrete findings, unresolved gaps, scope limitations, and sibling isolation. A shared conversational tribunal was rejected because sibling influence would break independent evaluation.
4. **Keep the established 100-point rubric stable.** The six-dimension matrix remains comparable to the earlier public report. Stars are refreshed but remain zero-weight context. A candidate's points or crown change only when primary evidence invalidates the prior feature assessment.
5. **Repair runtime only on reproduced failure.** Generated criticism alone does not authorize speculative changes. A failing supported surface or test must be reproduced first, then fixed with the smallest scoped edit.
6. **Fail closed on evidence and publication.** Missing NotebookLM authentication, incomplete judge output, inconsistent CSV/report facts, a failed clean install, a dirty post-commit tree, remote SHA drift, or an unretrievable SHA-pinned blob prevents a positive completion claim.

## Risks / Trade-offs

- **NotebookLM can reject new sources at the 600-source limit, while the account can reject notebook creation at 500 notebooks** → Reuse the canonical notebook's explicit processed-source subset, retain both failed attempts as evidence, and do not delete or repurpose unrelated user data.
- **NotebookLM may normalize code or fabricate implementation details** → Add a contradiction query and resolve every code claim against the checked-out source and executable tests.
- **Grok or `agy` may fail, hallucinate, inspect unrelated files, or omit required fields** → Use read-only, bounded prompts with an explicit output schema; retain service/quota failures; use only the brief-authorized fallback; and reject incomplete responses rather than silently repairing them.
- **Three accepted calls from one model family are isolated but not cross-provider independent** → Claim only process isolation and the exact provider/model provenance observed.
- **Stars and remote metadata can change mid-run** → Freeze one UTC snapshot and keep matrix, report, and judge packet consistent with it.
- **A high feature score can be mistaken for runtime truth** → State separately what the project-authored rubric, local structural backend, live IDR, and external judges do and do not prove.
- **Push may succeed while the handed-off URL targets mutable branch content** → Verify local full SHA equals `origin/master`, then retrieve the SHA-pinned blob/raw content before reporting the link.
