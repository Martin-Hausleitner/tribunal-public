## Context

The project already ships a dependency-free Python Tribunal core, a reusable skill, nine validated personas including a disclosed Karpathy-inspired critic, deterministic gates, a comparison matrix, and a public live-IDR report. Earlier evidence proves earlier snapshots only. This pass must re-observe mutable external systems while preserving the distinction between NotebookLM research, external judge opinions, executable package behavior, and project-authored comparative scoring.

The weekly guard is above its 50% threshold, so this pass cannot start new expensive Codex/OMX team bursts. The brief explicitly permits `agy` perspectives instead of Grok workers; three fresh `agy` sessions therefore satisfy the live tribunal path without violating the resource guard.

## Goals / Non-Goals

**Goals:**

- Produce a fresh, authenticated, source-grounded NotebookLM ledger with canonical identity, source state, three role queries, and one contradiction control.
- Reconfirm current OSS metadata from primary GitHub surfaces and keep scoring fully reproducible.
- Freeze a conclusion-free evidence packet before three isolated role judgments, then synthesize only accepted outputs.
- Re-run the real package through source, build, clean-install console, installed API, success, and expected-error paths.
- Publish internally consistent report/evidence artifacts and prove the exact committed report is public at an immutable blob URL.

**Non-Goals:**

- Claim that the deterministic local backend performs semantic fact-checking, visual UX inspection, NotebookLM retrieval, or cross-provider independence.
- Turn mutable star counts into quality points or legal conclusions.
- Impersonate Andrej Karpathy or attribute synthetic persona output to him.
- Add a new orchestration framework, browser UI, persistent quota authority, or tracing platform when established OSS can be composed.

## Decisions

1. **Use one canonical NotebookLM notebook and selected-source queries.** Reusing the verified canonical notebook preserves research continuity, while a fresh uniquely named source bundle and query ledger prove this pass occurred. Selected primary/current sources reduce contamination from historical duplicate pages. Creating another notebook was rejected because it fragments provenance without improving the brief's evidence.
2. **Prefer primary and executable evidence over generated prose.** NotebookLM and judge answers are retained as opinions with grounding metadata; repository reads, CLI output, wheel contents, GitHub API responses, and exact remote/blob checks control factual conflicts. Accepting model prose as self-validating evidence was rejected.
3. **Use three isolated `agy` sessions.** The resource guard forbids new expensive Codex/OMX bursts, and the brief allows `agy` judges. Each receives the same conclusion-free packet plus a role-specific rubric, no sibling output, and read-only scope. A response is accepted only if it identifies inspected evidence, score, recommendation, findings, gaps, and scope limitations.
4. **Refresh rather than redesign the OSS matrix.** The existing six-dimension 100-point rubric remains stable so score changes reflect evidence, not rubric drift. Current stars and repository facts are refreshed from canonical GitHub endpoints; stars remain zero-weight. A candidate, score, or crown changes only when primary evidence warrants it.
5. **Repair implementation only on reproduced defect.** Existing source is not rewritten merely because a generated judge suggests a flaw. A concrete failing test or manual surface observation is required, followed by the smallest scoped fix.
6. **Treat publication as fail-closed.** Unit, compile, build, skill, CSV, report, strict OpenSpec, evidence-link, clean-install, Git, remote SHA, and public blob checks must all agree. A working branch URL is insufficient; handoff uses the exact full commit SHA.

## Risks / Trade-offs

- **NotebookLM corpus contains historical duplicates or failed sources** → Query a selected current/primary subset, record counts and grounding IDs, and manually control contradictions.
- **External judge hallucinates or reads outside scope** → Require file/scope attestations, retain raw output, exclude unsupported verdicts, and retry only with a materially tighter prompt.
- **Three `agy` sessions are not statistical independence** → Claim only fresh session isolation and the provider/model provenance actually observed.
- **GitHub stars or repository metadata change during the run** → Timestamp one completed snapshot and make CSV/report match that frozen record.
- **A high project-authored matrix score is confused with runtime truth** → Keep comparative rubric, local structural score, and live semantic evidence explicitly separate throughout the report.
- **Public push succeeds but immutable artifact differs** → Compare local full SHA to remote branch SHA and retrieve the SHA-pinned raw/blob content before handoff.

## Migration Plan

1. Capture the starting SHA, clean status, tool/authentication state, and guard state.
2. Complete and strictly validate this OpenSpec change before live synthesis.
3. Add the pass-specific evidence source to the canonical notebook, wait for processing, then run and record the four selected-source queries.
4. Refresh GitHub metadata and freeze a conclusion-free judge packet.
5. Run three isolated role judgments, validate them, and synthesize agreements, disagreements, and controlling limitations.
6. Update report, matrix/snapshot only if evidence changed, and write E2E proof.
7. Run all gates; if a new defect appears, fix minimally and repeat only affected checks.
8. Commit, push, verify remote equality, and retrieve the immutable report blob. Rollback is a normal revert of this verification commit; no runtime data migration is involved.
